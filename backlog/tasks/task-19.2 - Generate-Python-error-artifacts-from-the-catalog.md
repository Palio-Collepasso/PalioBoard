## Description

<!-- SECTION:DESCRIPTION:BEGIN -->
Generate committed Python-side error artifacts from the validated catalog so backend code can import stable error identifiers and metadata instead of duplicating them by hand. The generator should follow the existing `python -m palio.app.*` CLI pattern and support consistent RFC 9457 `application/problem+json` response construction from catalog entries without moving endpoint-routing decisions into the catalog.
<!-- SECTION:DESCRIPTION:END -->

## Acceptance Criteria
<!-- AC:BEGIN -->
- [x] #1 The repository has committed per-module Python error-code artifacts under `apps/api/src/palio/modules/<module>/error_codes_generated.py`, each containing stable codes and catalog-derived metadata for that module only.
- [x] #2 Handwritten module `errors.py`, handwritten HTTP exception mapping, and any handwritten domain-specific helper constructors can consume the generated metadata consistently; these runtime and domain-specific pieces are not generated by this task.
- [x] #3 The task adds deterministic tests, documentation updates, and repo integration so the Python generation workflow participates in the shared errors-only and broader api/combined make workflows without deciding which endpoints return which errors or crossing module boundaries.
<!-- AC:END -->

## Implementation Plan

<!-- SECTION:PLAN:BEGIN -->
1. Wait for the validated merged catalog model from `TASK-19.1`, and confirm the active catalog contract is free of legacy `replaced_by` references before generating anything. The prerequisite output for this task is a module-aligned catalog with preserved owning-module provenance.
2. Add a Python generator CLI that follows the existing `python -m palio.app.*` pattern and consumes the merged catalog model from `TASK-19.1` rather than reparsing fragment files itself.
3. Emit committed per-module Python artifacts beside the backend modules they belong to: `apps/api/src/palio/modules/<module>/error_codes_generated.py`. Each generated file should contain stable symbolic codes plus catalog-derived metadata for that module, such as type URIs, recommended statuses, retry/exposure metadata, and translation keys, without generating exception classes or domain constructors.
4. Keep `apps/api/src/palio/modules/<module>/errors.py` handwritten and module-owned. Those handwritten files should be able to import their local `error_codes_generated.py`, define module exception classes, and provide any domain-specific helper constructors manually. HTTP exception mapping should remain handwritten in the API layer or another explicit runtime seam, not generated here.
5. Avoid a monolithic shared generated Python registry as the primary backend surface. Shared runtime utilities may exist as thin handwritten helpers if needed, but the generator’s main contract is per-module `error_codes_generated.py` output aligned with backend boundaries.
6. Add deterministic unit tests in `apps/api/tests/unit/` that verify CLI export behavior, stable generated file contents, per-module placement, and faithful metadata emission from the catalog. Include drift checks that compare regenerated module files against the committed files.
7. Update repo integration and docs: extend the grouped workflow commands so the Python generator participates in the errors, api, and combined contract paths where appropriate; update `docs/api/README.md`, `docs/api/errors/README.md`, and any affected local-dev workflow docs to describe the per-module generated file convention and the handwritten/runtime ownership split.
8. Validation checkpoints for implementation: `make test-api-unit`, `make lint`, `make typecheck`, `make check-openapi` when relevant, and the grouped errors workflow once it exists. The Python generator should stay deterministic across repeated runs and should not decide which endpoints return which errors.
<!-- SECTION:PLAN:END -->
